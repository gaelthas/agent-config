# 设计文档模式

标准化设计文档模板，确保团队输出风格一致、内容完整的设计文档。

## 何时激活

- 启动新功能或项目
- 技术评审准备
- 架构决策记录
- 前后端接口约定
- 数据库设计评审

## PRD 模板要点

### 必备章节

1. **需求背景** — 解释"为什么做"
2. **目标用户** — 明确"给谁做"
3. **功能清单** — 列出"做什么"，标注优先级（P0/P1/P2）
4. **用户故事** — 用 "作为…我希望…以便…" 格式
5. **详细需求** — 流程图 + 业务规则 + 异常场景
6. **非功能需求** — 性能、安全、兼容性
7. **验收标准** — 可量化、可测试的检查清单

### 常见错误

- ❌ 只写功能列表，不写业务背景
- ❌ 验收标准模糊（如"性能要好"）
- ❌ 遗漏异常场景和边界条件
- ✅ 每个验收标准可被自动化测试验证

## 技术方案模板要点

### 必备章节

1. **方案概述** — 一段话概括需求和技术选型
2. **架构设计** — 系统架构图 + 模块划分
3. **接口设计** — API 列表 + 详细定义
4. **数据模型** — ER 图 + 表结构
5. **实现计划** — 开发步骤 + 风险评估
6. **测试策略** — 覆盖范围 + 重点场景

### 技术选型表格

```markdown
| 组件 | 选择 | 备选 | 理由 |
|------|------|------|------|
| Web 框架 | Gin | Fiber | 社区成熟度高、文档完善 |
| 数据库 | MySQL 8 | PostgreSQL | 团队熟悉度 |
| 缓存 | Redis | Memcached | 支持更多数据结构 |
```

## 架构决策记录 (ADR)

### ADR 模板

```markdown
# ADR-XXX: [决策标题]

## 背景
[为什么要做这个决策]

## 决策
[做了什么决策]

## 后果
### 优点
- [好处]

### 缺点
- [代价]

### 考虑的替代方案
- [方案A]：[优缺点]
- [方案B]：[优缺点]

## 状态
[提议 / 已接受 / 已弃用 / 已替代]

## 日期
[YYYY-MM-DD]
```

## 接口文档规范

### 端点定义

```markdown
### POST /api/v1/users

#### 说明
创建新用户

#### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 是 | 用户名，1-50 字符 |
| email | string | 是 | 邮箱地址 |
| phone | string | 否 | 手机号 |

#### 请求示例
{json 示例}

#### 响应示例
{json 示例}

#### 错误码
| 状态码 | 错误码 | 说明 |
|--------|--------|------|
| 400 | validation_error | 参数验证失败 |
| 409 | duplicate_email | 邮箱已存在 |
```

## 数据模型文档规范

### 表定义格式

```markdown
### users 表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| name | VARCHAR(50) | NOT NULL | 用户名 |
| email | VARCHAR(100) | NOT NULL, UNIQUE | 邮箱 |
| created_at | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**索引：**
- `idx_users_email` (email) — 登录查询
- `idx_users_created_at` (created_at) — 时间排序
```

## 文档质量检查清单

- [ ] 文档标题和日期完整
- [ ] 所有技术选型有理由
- [ ] 接口示例覆盖成功和错误场景
- [ ] 数据模型包含索引设计
- [ ] 风险评估有缓解措施
- [ ] 无 TODO 或占位符残留
